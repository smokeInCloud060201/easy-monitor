# Plan 3: APM Trace Search & Error Tracking

<objective>
Replace the basic trace-ID-only search with a Datadog-style trace search page featuring filters (service, resource, status, duration range), paginated results, and enhanced trace detail view showing span metadata, error info, and resource attribution.
</objective>

## Tasks

<task id="3.1">
<title>Rebuild Trace Explorer with filter-based search</title>
<read_first>
- dashboard/src/pages/TraceExplorer.tsx
- dashboard/src/lib/api.ts
</read_first>
<action>
Rewrite `TraceExplorer.tsx` to replace the single trace-ID input with a full search interface:

1. **Filter Bar** (horizontal row at top):
   - Service dropdown: populated by calling existing `GET /api/v1/apm/services`
   - Resource dropdown: populated after service is selected via `GET /api/v1/apm/services/:name/resources`
   - Status toggle: All / OK / Error (buttons/pills)
   - Min Duration input (ms)
   - Max Duration input (ms)
   - Search button
   - Keep the trace ID direct lookup as a secondary input (collapsible "Search by Trace ID" section)

2. **Results Table**:
   - Columns: Trace ID (first 12 chars, monospace), Root Service, Root Operation, Duration (ms, with bar visualization), Spans, Status (green/red badge), Timestamp
   - Each row clickable → navigates to `/traces/:traceId`
   - Sort by clicking column headers (duration, timestamp)
   - Pagination: "Load More" button or page numbers

3. **Auto-search on page load**: If no filters, show latest 50 traces
4. **URL state**: Store filters in query params so links are shareable

Call `searchTraces(filters)` from `api.ts` for all searches.
</action>
<acceptance_criteria>
- `TraceExplorer.tsx` has filter bar with service dropdown, resource dropdown, status toggle, duration range inputs
- Results displayed in a table with columns: Trace ID, Service, Operation, Duration, Spans, Status, Timestamp
- Clicking a row navigates to `/traces/:traceId`
- Auto-loads latest traces on mount
- Pagination support (limit/offset)
- Trace ID direct lookup still available
</acceptance_criteria>
</task>

<task id="3.2">
<title>Enhance Trace Detail with span metadata panel</title>
<read_first>
- dashboard/src/pages/TraceDetail.tsx
- dashboard/src/components/traces/SpanWaterfall.tsx
</read_first>
<action>
Enhance `TraceDetail.tsx`:

1. **Trace Header Enhancement**: Show root span service + operation, total duration, span count, error status badge, timestamp
2. **Span Metadata Panel**: When a span is clicked in the waterfall, show a right-side slide-out panel with:
   - Span name, service, resource
   - Duration (ms) 
   - Start time, end time
   - Error status (with error message if present)
   - Span ID, Parent ID
3. **Error Highlight**: Spans with `error > 0` should have a red left border and red background tint in the waterfall
4. **Service Color Coding**: Assign consistent colors to services in the waterfall (use a hash-based color palette)

The existing split-pane layout (60% waterfall, 40% logs) stays. The metadata panel overlays on the waterfall area.
</action>
<acceptance_criteria>
- Trace header shows root service, operation, total duration, span count
- Clicking a span shows metadata panel with name, service, duration, timestamps, error info
- Error spans have red visual treatment in waterfall
- Services have consistent color coding
</acceptance_criteria>
</task>
