# Plan 1: APM Backend Foundation

<objective>
Replace mock trace APIs with real ClickHouse queries, add time-series RED metrics storage, and create the backend endpoints needed for Datadog-style APM service/resource detail pages.
</objective>

## Tasks

<task id="1.1">
<title>Add RED metrics time-series table to ClickHouse</title>
<read_first>
- master-service/src/storage/clickhouse.rs
</read_first>
<action>
In `initialize_clickhouse()`, add a new CREATE TABLE statement after the existing ones:

```sql
CREATE TABLE IF NOT EXISTS easy_monitor_red_metrics (
    service String,
    resource String,
    timestamp Int64,
    requests Float64,
    errors Float64,
    duration_sum Float64,
    duration_avg Float64,
    duration_p50 Float64,
    duration_p95 Float64,
    duration_p99 Float64,
    count UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(toDateTime(timestamp / 1000))
ORDER BY (service, resource, timestamp)
TTL toDateTime(timestamp / 1000) + INTERVAL 30 DAY;
```

Execute the CREATE TABLE in `initialize_clickhouse()` alongside existing tables.
</action>
<acceptance_criteria>
- `clickhouse.rs` contains `easy_monitor_red_metrics` CREATE TABLE statement
- `initialize_clickhouse()` sends the new CREATE TABLE query to ClickHouse
- Table has columns: service, resource, timestamp, requests, errors, duration_sum, duration_avg, duration_p50, duration_p95, duration_p99, count
</acceptance_criteria>
</task>

<task id="1.2">
<title>Extend RED metrics engine to persist time-series to ClickHouse</title>
<read_first>
- master-service/src/processors/trace_metrics.rs
- master-service/src/storage/clickhouse.rs
</read_first>
<action>
Modify `start_trace_metrics_engine` to also batch-insert RED aggregates into ClickHouse every 10s window:

1. Add a `reqwest::Client` parameter or create one internally
2. In the 10-second interval tick, after computing per-service:resource aggregates:
   - For each unique (service, resource) pair, collect all individual span durations in a `Vec<f64>` (need to accumulate these alongside the existing counters)
   - Compute percentiles: sort durations, p50 = durations[len*50/100], p95 = durations[len*95/100], p99 = durations[len*99/100]
   - Build JSONEachRow payload with fields: service, resource, timestamp (current epoch ms), requests (count of spans), errors, duration_sum, duration_avg, duration_p50, duration_p95, duration_p99, count
   - POST to ClickHouse: `INSERT INTO easy_monitor_red_metrics FORMAT JSONEachRow`
3. Keep existing event bus metric publishing unchanged (in-memory metrics still power the Service Catalog)

Key change: Instead of just incrementing counters in DashMap, store individual durations per key in a `DashMap<String, Vec<f64>>` for percentile computation. Use a separate `DashMap<String, (f64, f64)>` for (rate, error) counts.
</action>
<acceptance_criteria>
- `trace_metrics.rs` creates a reqwest::Client for ClickHouse writes
- Each 10s tick inserts rows into `easy_monitor_red_metrics` via ClickHouse HTTP API
- Percentile computation exists: p50, p95, p99 from sorted duration vectors
- Existing in-memory metric publishing to event bus is unchanged
</acceptance_criteria>
</task>

<read_first>

<action>
Add new types and handler in `queries.rs`:

```rust
#[derive(Deserialize)]
pub struct TraceSearchRequest {
    pub service: Option<String>,
    pub resource: Option<String>,
    pub status: Option<String>,       // "ok" | "error"
    pub min_duration_ms: Option<f64>,
    pub max_duration_ms: Option<f64>,
    pub limit: Option<usize>,
    pub offset: Option<usize>,
}

#[derive(Serialize)]
pub struct TraceSummary {
    pub trace_id: String,
    pub root_service: String,
    pub root_name: String,
    pub duration_ms: f64,
    pub span_count: u64,
    pub error: bool,
    pub timestamp: String,
}

#[derive(Serialize)]
pub struct TraceSearchResponse {
    pub traces: Vec<TraceSummary>,
    pub total: u64,
}
```

Handler `search_traces`:
1. Build ClickHouse query against `easy_monitor_traces`:
   - Group by trace_id to get root span info: `SELECT trace_id, argMin(service, timestamp) as root_service, argMin(name, timestamp) as root_name, max(duration)/1000.0 as duration_ms, count() as span_count, max(error) as has_error, min(timestamp) as ts FROM easy_monitor_traces WHERE 1=1`
   - Add WHERE clauses for service, resource, error status, duration filters
   - Add `GROUP BY trace_id ORDER BY ts DESC LIMIT {limit} OFFSET {offset}`
2. Parse response into `Vec<TraceSummary>`
3. Return with total count

Register in `mod.rs`:
```rust
.route("/traces/search", post(queries::search_traces))
```
</action>

```rust
    pub p99_duration_ms: f64,

pub struct RedTimePoint {
    pub timestamp: i64,
    pub requests: f64,
    pub errors: f64,
    pub avg_duration: f64,
    pub p95_duration: f64,
    pub p99_duration: f64,
}
```

Query `easy_monitor_red_metrics` for time-series data:
```sql
SELECT timestamp, sum(requests) as requests, sum(errors) as errors,
       avg(duration_avg) as avg_duration, max(duration_p95) as p95_duration,
       max(duration_p99) as p99_duration
FROM easy_monitor_red_metrics
WHERE service = '{name}' AND timestamp > {now - 1h in ms}
GROUP BY timestamp ORDER BY timestamp ASC FORMAT JSON
```

**Enhanced Resources** (`GET /apm/services/:name/resources`):
Modify existing `get_resources` to also return RED metrics per resource by querying the latest window from `easy_monitor_red_metrics`:
