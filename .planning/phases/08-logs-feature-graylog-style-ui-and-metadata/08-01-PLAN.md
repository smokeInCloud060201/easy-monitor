# Plan 1: Logs Backend — Rich Metadata API & ClickHouse Schema Enhancement

<objective>
Extend the log data pipeline and API to support GrayLog-style rich metadata: structured fields, log-level filtering, timestamp-based histogram aggregation, field statistics, and expandable log detail view with all metadata fields. This replaces the current minimal (trace_id, service, message) API response with full structured log data.
</objective>

## Tasks

<task id="1.1">
<title>Extend ClickHouse logs schema with metadata columns</title>
<read_first>
- master-service/src/storage/clickhouse.rs
</read_first>
<action>
Modify `initialize_clickhouse()` in `clickhouse.rs` — replace the existing `easy_monitor_logs` CREATE TABLE with an ALTER TABLE migration that adds new columns (idempotent), OR create a new table and keep the old one as fallback. The recommended approach is to use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for each new column:

```sql
ALTER TABLE easy_monitor_logs ADD COLUMN IF NOT EXISTS span_id String DEFAULT '';
ALTER TABLE easy_monitor_logs ADD COLUMN IF NOT EXISTS host String DEFAULT '';
ALTER TABLE easy_monitor_logs ADD COLUMN IF NOT EXISTS source String DEFAULT '';
ALTER TABLE easy_monitor_logs ADD COLUMN IF NOT EXISTS attributes String DEFAULT '{}';
```

Also update the CREATE TABLE statement for fresh installs to include all columns:

```sql
CREATE TABLE IF NOT EXISTS easy_monitor_logs (
    trace_id String,
    span_id String,
    service String,
    level String,
    message String,
    pod_id String,
    namespace String,
    node_name String,
    host String,
    source String,
    attributes String DEFAULT '{}',
    timestamp Int64
) ENGINE = MergeTree()
ORDER BY (service, timestamp);
```

The `attributes` column stores a JSON string of arbitrary key-value metadata pairs — mirroring GrayLog's extensible field model.
</action>
<acceptance_criteria>
- `clickhouse.rs` contains ALTER TABLE statements for `span_id`, `host`, `source`, `attributes` columns
- CREATE TABLE statement includes all new columns with defaults
- `initialize_clickhouse()` executes the ALTER TABLE statements after CREATE TABLE
</acceptance_criteria>
</task>

<task id="1.2">
<title>Enhance ClickHouse writer to persist all metadata fields</title>
<read_first>
- master-service/src/storage/clickhouse.rs
- shared-proto/proto/logs.proto
</read_first>
<action>
In `start_clickhouse_writer`, update the logs batch flush to include all new fields from the LogEntry proto. The proto already has a `tags` map — extract additional fields from it:

```rust
for log in logs_batch.drain(..) {
    let pod_id = log.tags.get("pod_id").cloned().unwrap_or_default();
    let namespace = log.tags.get("namespace").cloned().unwrap_or_default();
    let node_name = log.tags.get("node_name").cloned().unwrap_or_default();
    let host = log.tags.get("host").cloned().unwrap_or_default();
    let source = log.tags.get("source").cloned().unwrap_or_default();
    let span_id = log.tags.get("span_id").cloned().unwrap_or_default();
    
    // Build attributes JSON from remaining tags (exclude extracted ones)
    let known_keys = ["pod_id", "namespace", "node_name", "host", "source", "span_id"];
    let extra_attrs: serde_json::Map<String, serde_json::Value> = log.tags.iter()
        .filter(|(k, _)| !known_keys.contains(&k.as_str()))
        .map(|(k, v)| (k.clone(), serde_json::Value::String(v.clone())))
        .collect();
    let attributes_json = serde_json::to_string(&extra_attrs).unwrap_or_default();

    let row = json!({
        "trace_id": log.trace_id,
        "span_id": span_id,
        "service": log.service,
        "level": log.level,
        "message": log.message,
        "pod_id": pod_id,
        "namespace": namespace,
        "node_name": node_name,
        "host": host,
        "source": source,
        "attributes": attributes_json,
        "timestamp": log.timestamp,
    });
    payload.push_str(&row.to_string());
    payload.push('\n');
}
```

No proto changes needed — the existing `tags` map can carry all metadata.
</action>
<acceptance_criteria>
- ClickHouse writer extracts `host`, `source`, `span_id` from `log.tags`
- Remaining unknown tags are serialized as JSON into `attributes` column
- Insert statement writes all 12 columns to `easy_monitor_logs`
</acceptance_criteria>
</task>

pub struct LogsQueryRequest {

Add a new endpoint `POST /api/v1/logs/histogram` for the GrayLog-style timeline histogram:

```rust
#[derive(Deserialize)]
pub struct LogHistogramRequest {
    pub service: Option<String>,
    pub level: Option<String>,
    pub keyword: Option<String>,
    pub from_ts: Option<i64>,
    pub to_ts: Option<i64>,
    pub interval: Option<String>,   // "minute" | "hour" | "day", default "minute"
}

#[derive(Serialize)]
pub struct HistogramBucket {
    pub timestamp: i64,
    pub count: u64,
    pub error_count: u64,
    pub warn_count: u64,
}

#[derive(Serialize)]
pub struct LogHistogramResponse {
    pub buckets: Vec<HistogramBucket>,
}
```

Handler `log_histogram`:
1. Determine bucket size — map interval to ClickHouse `toStartOfMinute`/`toStartOfHour`/`toStartOfDay`
2. Build query:
```sql
SELECT 
    toUnixTimestamp(toStartOfMinute(toDateTime(timestamp / 1000))) * 1000 as bucket,
    count() as count,
    countIf(level = 'ERROR') as error_count,
    countIf(level = 'WARN') as warn_count
FROM easy_monitor_logs
WHERE timestamp >= {from_ts} AND timestamp <= {to_ts}
{AND service = '...' if provided}
{AND message ILIKE '%keyword%' if provided}
GROUP BY bucket
ORDER BY bucket ASC
FORMAT JSON
```
3. Parse and return as `Vec<HistogramBucket>`
4. Include a mock fallback generating 60 buckets of random data

Register in `mod.rs`:
```rust
.route("/logs/histogram", post(queries::log_histogram))
```
</action>

- `log_histogram` handler queries ClickHouse with time-bucket aggregation
- Response includes `count`, `error_count`, `warn_count` per bucket
- Supports minute/hour/day interval granularity
